# MySQL 8.4 Stage 1 Batch 3

## Общий префикс

```text
Работай строго по:
- AGENTS.md
- docs/architecture/current-architecture.md
- docs/architecture/current-state.md
- docs/architecture/change-checklist.md
- docs/architecture/mysql-84-target-architecture.md
- docs/architecture/mysql-84-migration-plan.md
- docs/business-rules/*.md

Важно:
- Это MySQL 8.4 Stage 1: MySQL Platform and Operations Baseline.
- Этот batch про backup/restore/runbook, не про domain SQL cutover.
- Нельзя менять production/VDS без отдельного явного запроса.
- Нельзя считать SQL dump достаточным backup, если файлы карточек остаются в
  filesystem/object storage.
- Учитывай результаты Stage 1 Batch 1 audit:
  - current file storage root resolves from `TSPCC_STORAGE_DIR` or
    `storage/cards`;
  - current data root resolves from `TSPCC_DATA_DIR` or `data/database.json`;
  - Stage 0 file baseline: `storage/cards/<qrId>` dirs and attachment metadata
    must be reconciled together;
  - attachment `size` is present in current metadata, checksum/hash is not
    present and must be generated by backup/reconciliation tooling or reported
    as unavailable.
```

## Промт

```text
Нужно выполнить Stage 1 Batch 3: подготовить production-grade backup/restore
runbook for MySQL + card files.

Цель:
- зафиксировать проверяемый backup set:
  SQL dump + file storage backup + manifest;
- подготовить restore rehearsal path for test environment;
- определить RPO/RTO baseline.

Что сделать:
1. Добавить/обновить docs/scripts for:
   - mysqldump logical backup;
   - file storage archive/snapshot;
   - manifest generation;
   - restore rehearsal.
   Operational scripts should live under `scripts/mysql/` or another explicitly
   documented Stage 1 operations folder. They must be non-domain and must not
   change application source of truth.
2. Manifest должен включать:
   - timestamp;
   - app version/git commit;
   - schema migration version placeholder;
   - domain counts placeholder;
   - SQL dump filename/size;
   - file archive filename/size;
   - file count/checksum summary;
   - source `TSPCC_DATA_DIR` / `TSPCC_STORAGE_DIR` or explicit paths used.
3. Описать retention baseline:
   - minimum daily full logical dump;
   - matching file storage archive;
   - retention not less than 14 days unless a later operations decision changes
     it.
4. Описать restore verification:
   - SQL restored;
   - files restored;
   - file metadata matches physical files.
5. Описать RPO/RTO baseline from Batch 1 audit:
   - initial RPO: 24h until a stricter operational schedule is approved;
   - initial RTO target: 2-4h until rehearsal gives measured numbers.
6. Описать команды, которые в следующих шагах потребуют approval/network or
   environment access:
   - installing MySQL 8.4;
   - installing MySQL client tools if missing;
   - running backup/restore against VDS/staging;
   - copying backup archives outside the production disk.

Что нельзя делать:
- не менять production data;
- не запускать destructive restore outside test env;
- не добавлять real secrets;
- не начинать Stage 2.
- не считать backup complete без file storage archive and manifest;
- не использовать Git history or JSON export as replacement for SQL+files
  restore point.

Формат ответа:

Ответ по итогам batch ОБЯЗАТЕЛЬНО выводи на русском языке; технические статусы `PASS` / `FAIL` / `BLOCKED`, имена команд, маршрутов, файлов и таблиц не переводить.
1. Какие backup/restore artifacts добавлены.
2. Как формируется backup set.
3. Как выполняется restore rehearsal.
4. Какие проверки выполнены.
5. Остаточные операционные риски.
6. Какие approval/network-dependent commands оставлены на ручной/local-test
   stage.
```

## Ручная проверка после Prompt

Если есть локальная/test MySQL и test file storage, выполнить documented
backup/restore rehearsal. Production не трогать.
