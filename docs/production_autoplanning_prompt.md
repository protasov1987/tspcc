# Промт для внедрения функции автоматического планирования операций

Ниже собрана единая инструкция для реализации автоматического планирования на базе уже существующего сменного планировщика. Следуй этому промту буквально. Если какое-либо правило не описано в коде, но описано здесь, источником истины считается этот документ.

## Контекст и обязательная совместимость
- Не ломать существующее ручное планирование через `#production-shift-plan-modal`.
- Использовать текущие сущности приложения:
  - `cards`
  - `card.operations`
  - `productionShiftTasks`
  - `productionShifts`
  - `productionShiftTimes`
  - `areas`
  - `productionSchedule`
- Не менять текущую базовую логику доступности смен:
  - планировать можно только в смены, которые не `LOCKED`, не `CLOSED` и не `isFixed`;
  - при открытии автопланирования карта должна быть только в статусе `PROVIDED` или `PLANNING`.
- Не менять текущую базовую логику расчёта остатка по операции:
  - для quantity-driven операций использовать ту же фактическую базу, что и `getOperationPlanningSnapshot(...)`;
  - для остальных операций использовать минуты.
- Для единиц планирования использовать только текущие обозначения:
  - обычная операция: `изд`
  - sample-операция `WITNESS`: `ОС`
  - sample-операция `CONTROL`: `ОК`

## Цель функции
- Добавить автоматическое планирование, которое умеет создавать плановые задания по одной карте сразу на несколько операций и несколько смен вперёд.
- Автопланирование должно работать в двух режимах:
  - полное планирование: построить план на весь фактический остаток выбранного диапазона операций;
  - частичное планирование: построить план только на часть остатка по количеству или по горизонту смен.
- Автопланирование не должно подменять ручной режим. Пользователь должен иметь возможность:
  - продолжать вручную добавлять отдельные операции в смену;
  - запускать автопланирование как отдельное действие в том же модальном окне.

## Обязательные UI-изменения модального окна
- В существующее модальное окно `#production-shift-plan-modal` добавить отдельный блок настроек автопланирования `#production-shift-auto-plan-settings`.
- В блоке автопланирования обязательно должны быть следующие поля:
  - `#production-shift-auto-plan-enabled`
    Назначение: переключение между ручным и автоматическим режимом внутри модального окна.
  - `#production-shift-auto-plan-mode`
    Значения:
    - `FULL_ROUTE` — планировать все доступные операции карты от первой незапланированной до последней доступной;
    - `FROM_SELECTED` — планировать от выбранной операции до конца маршрута;
    - `BETWEEN_SELECTED` — планировать только диапазон от операции `startOpId` до операции `endOpId`;
    - `SINGLE_OPERATION` — автопланировать только одну выбранную операцию по нескольким сменам.
  - `#production-shift-auto-plan-volume-mode`
    Значения:
    - `FULL` — планировать весь остаток;
    - `PARTIAL_QTY` — планировать только заданное количество `изд/ОС/ОК`;
    - `PARTIAL_MINUTES` — планировать только заданный объём в минутах, если операция не quantity-driven.
  - `#production-shift-auto-plan-qty`
    Назначение: количество для частичного планирования в единице `изд/ОС/ОК`.
    Правило: поле активно только в режиме `PARTIAL_QTY`.
  - `#production-shift-auto-plan-minutes`
    Назначение: количество минут для частичного планирования non-qty операций.
    Правило: поле активно только в режиме `PARTIAL_MINUTES`.
  - `#production-shift-auto-plan-horizon-shifts`
    Назначение: максимум смен вперёд, в которые разрешено строить план, считая текущую стартовую смену.
    Правило: значение целое, минимум `1`.
  - `#production-shift-auto-plan-delay-minutes`
    Назначение: обязательная задержка между окончанием предыдущей операции и разрешением запуска следующей.
    Правило: значение целое, минимум `0`.
  - `#production-shift-auto-plan-transfer-batch-qty`
    Назначение: минимальная передаточная партия на следующую операцию.
    Правило: значение целое, минимум `1`.
  - `#production-shift-auto-plan-tail-batch-mode`
    Значения:
    - `ALLOW_LAST_PARTIAL_BATCH` — остаток меньше партии можно передавать последним неполным пакетом;
    - `ONLY_FULL_BATCHES` — передавать только полные партии, хвост остаётся непереданным.
  - `#production-shift-auto-plan-area-mode`
    Значения:
    - `CURRENT_AREA_ONLY` — использовать только участок, из которого открыта модалка;
    - `AUTO_ALLOWED_AREAS` — автоматически искать ближайший доступный участок из `allowedAreaIds` операции.
  - `#production-shift-auto-plan-split-mode`
    Значения:
    - `ALLOW_SPLIT_BY_SHIFTS` — одну операцию можно разбивать на несколько смен;
    - `SINGLE_SHIFT_ONLY` — задание по операции должно целиком помещаться в одну смену, иначе слот пропускается.
  - `#production-shift-auto-plan-priority-mode`
    Значения:
    - `EARLIEST_SHIFT_FIRST` — сначала ближайшие смены;
    - `CURRENT_SHIFT_FILL_FIRST` — сначала максимально заполнить текущую стартовую смену, затем следующие.
  - `#production-shift-auto-plan-start-op`
    Назначение: стартовая операция для режима `BETWEEN_SELECTED`.
  - `#production-shift-auto-plan-end-op`
    Назначение: конечная операция для режима `BETWEEN_SELECTED`.
- В правой части модалки добавить предпросмотр результата автопланирования:
  - `#production-shift-auto-plan-preview`
  - в превью обязательно показывать:
    - сколько задач будет создано;
    - по каким операциям;
    - на какие даты/смены/участки;
    - сколько минут и сколько `изд/ОС/ОК` попадёт в каждую задачу;
    - какие операции остались непокрытыми и почему.
- В футере модалки добавить отдельную кнопку:
  - `#production-shift-auto-plan-run`
  - текст: `Построить автоплан`
- Кнопка `#production-shift-plan-save` должна продолжать работать как ручное добавление одной операции и не должна неожиданно запускать автоплан.

## Обязательные изменения модели данных
- Для корректной поддержки задержки между операциями недостаточно хранить только `date + shift + plannedPartMinutes`.
- В `productionShiftTasks` добавить и нормализовать новые поля:
  - `planningMode`: `MANUAL` или `AUTO`
  - `autoPlanRunId`: идентификатор одного запуска автопланирования
  - `plannedStartAt`: timestamp планового старта внутри смены
  - `plannedEndAt`: timestamp планового окончания внутри смены
  - `transferBatchQty`: размер передаточной партии, с которым была создана задача
  - `delayMinutes`: задержка до следующей операции, с которой была создана задача
  - `sourceShiftDate`: стартовая дата окна автопланирования
  - `sourceShift`: стартовая смена окна автопланирования
- Новые поля обязательны именно для auto-задач.
- Для старых и ручных задач обратная совместимость обязательна:
  - если поля отсутствуют, ручная задача должна продолжать работать как раньше.

## Базовые правила расчёта объёма операции
- Для quantity-driven операций:
  - исходный остаток берётся только из `getOperationPlanningSnapshot(cardId, routeOpId)`;
  - для автопланирования использовать только целые единицы `изд/ОС/ОК`;
  - дробные `0.25`, `0.5`, `0.75` для автопланирования запрещены;
  - `minutesPerUnit = baseMinutes / baseQty`;
  - минуты на задачу считаются так:
    - `taskQty` — целое число;
    - `taskMinutes = ceil(minutesPerUnit * taskQty)`.
- Для non-qty операций:
  - объём определяется только в минутах;
  - передаточная партия `transferBatchQty` и правила передачи поштучно на них не распространяются;
  - если выбран режим `PARTIAL_QTY`, а операция не quantity-driven, такую операцию в автоплан не включать и показать причину в превью.

## Явные правила полного автопланирования
- Полное планирование означает:
  - для каждой операции выбранного диапазона целевой объём равен полному незапланированному фактическому остатку на момент запуска;
  - для quantity-driven операций это `snapshot.uncoveredQty`;
  - для non-qty операций это `snapshot.availableToPlanMinutes`.
- Операция считается полностью покрытой автопланом только если:
  - для quantity-driven операций `remainingAutoQty == 0`;
  - для non-qty операций `remainingAutoMinutes == 0`.
- Если горизонт смен закончился раньше полного покрытия:
  - сохранять только реально размещённую часть;
  - в превью и в toast явно показывать, что план построен частично из-за нехватки сменовой ёмкости;
  - статус карты нельзя искусственно переводить в “полностью запланировано”, если остаток остался.

## Явные правила частичного автопланирования
- Частичное планирование допускается в двух формах:
  - по количеству `изд/ОС/ОК`;
  - по минутам.
- Режим `PARTIAL_QTY`:
  - пользователь задаёт `requestedQty`;
  - итоговый целевой объём операции:
    - `targetQty = min(requestedQty, snapshot.uncoveredQty)`;
  - если `targetQty <= 0`, задача по операции не создаётся.
- Режим `PARTIAL_MINUTES`:
  - пользователь задаёт `requestedMinutes`;
  - итоговый целевой объём операции:
    - `targetMinutes = min(requestedMinutes, snapshot.availableToPlanMinutes)`;
  - если операция quantity-driven, сначала вычислить максимально возможное целое количество изделий в пределах `requestedMinutes`, затем пересчитать минуты через `ceil(minutesPerUnit * qty)`;
  - если в пределах `requestedMinutes` не помещается хотя бы `1` единица, операция в автоплан не попадает.
- Частичный автоплан не имеет права:
  - создавать задачи на объём больше незапланированного остатка;
  - создавать quantity-driven задачи на дробную часть изделия/ОС/ОК;
  - создавать пустые задания с `0 мин`.

## Явные правила выбора операций
- В автоплан должны попадать только операции, которые уже разрешены текущим планировщиком:
  - исключить `material issue`;
  - исключить `material return`;
  - исключить `drying`;
  - исключить скрытые операции.
- Порядок операций всегда только по маршруту карты.
- Нарушать маршрутный порядок запрещено.
- Следующая операция не может получить объём раньше, чем этот объём станет доступен по правилам передачи с предыдущей операции.

## Явные правила передачи объёма на следующую операцию
- Для quantity-driven операций правило передачи обязательно и однозначно:
  - следующая операция может стартовать только после того, как на предыдущей операции накоплен доступный к передаче объём;
  - доступный к передаче объём считается не по факту выполнения, а по уже построенному автоплану в рамках текущего `autoPlanRunId`.
- Базовое правило передачи:
  - `releasedQty(opN)` = количество, которое уже размещено автопланом на операции `opN` и имеет `plannedEndAt`;
  - `consumedQty(opN+1)` = количество, уже переданное на следующую операцию в рамках этого же запуска;
  - `transferableQty = releasedQty(opN) - consumedQty(opN+1)`.
- Следующая операция получает право создать новую задачу только если:
  - `transferableQty >= transferBatchQty`, либо
  - `tail-batch-mode == ALLOW_LAST_PARTIAL_BATCH` и предыдущая операция полностью размещена, а `transferableQty > 0`.
- Размер одной передаваемой партии:
  - если доступно несколько полных партий и в смене хватает времени, можно размещать несколько партий подряд;
  - каждая создаваемая задача должна иметь явный `taskQty`;
  - `taskQty` не может быть больше:
    - `transferableQty`,
    - остатка по текущей операции,
    - доступной ёмкости выбранной смены.
- Для `ONLY_FULL_BATCHES`:
  - хвост меньше `transferBatchQty` запрещено передавать;
  - этот хвост остаётся непокрытым, пока пользователь не изменит настройки.
- Для non-qty операций правило передачи партиями не применяется.
- Если выбран режим `SINGLE_OPERATION`, правило передачи на следующую операцию не используется, потому что следующая операция не планируется в этом запуске.

## Явные правила задержки между операциями
- Задержка задаётся в минутах через `delayMinutes`.
- Для последовательных операций в одном `autoPlanRunId`:
  - `earliestStart(nextOp) = plannedEndAt(previousReleasedBatch) + delayMinutes * 60_000`.
- Новая задача следующей операции может быть поставлена только в тот слот, у которого:
  - `slotStartAt <= plannedStartAt < slotEndAt`;
  - `plannedStartAt >= earliestStart(nextOp)`.
- Если внутри текущей смены после задержки осталось времени меньше, чем нужно для минимальной партии:
  - слот считается непригодным;
  - планировщик переходит к следующей доступной смене.
- Если `delayMinutes = 0`, следующая операция может начинаться сразу после `plannedEndAt` предыдущей партии, но всё равно не раньше.

## Явные правила сменовой ёмкости
- Для совместимости с текущей ручной моделью ёмкость ячейки считать так же, как сейчас:
  - `cellCapacityMinutes = shiftDurationMinutes`;
  - количество сотрудников в ячейке не умножает ёмкость автоматически.
- Автоплан не должен вводить другую формулу ёмкости, чем ручной режим.
- Для каждого слота:
  - `busyMinutes = сумма plannedPartMinutes всех задач в этой ячейке`;
  - `freeMinutes = cellCapacityMinutes - busyMinutes`.
- Запрещено создавать задачу, если она выводит `freeMinutes` в отрицательное значение.

## Явные правила выбора слота
- Стартовым слотом считается ячейка, из которой открыта модалка.
- Далее слоты перебираются строго вперёд по времени.
- В режиме `CURRENT_SHIFT_FILL_FIRST`:
  - сначала максимально использовать стартовую смену;
  - затем переходить к следующим.
- В режиме `EARLIEST_SHIFT_FIRST`:
  - всегда брать самый ранний пригодный слот в пределах горизонта.
- В режиме `CURRENT_AREA_ONLY`:
  - использовать только `areaId` текущей модалки.
- В режиме `AUTO_ALLOWED_AREAS`:
  - для операции разрешены только участки из `allowedAreaIds`;
  - если список пустой, операция может планироваться на текущий участок;
  - если текущий участок не разрешён и других разрешённых участков в горизонте нет, операция остаётся непокрытой.

## Явные правила дробления задания
- В режиме `ALLOW_SPLIT_BY_SHIFTS`:
  - остаток операции можно разбивать на несколько задач по нескольким сменам.
- В режиме `SINGLE_SHIFT_ONLY`:
  - если полный объём конкретной партии не помещается в текущую смену, задача в эту смену не создаётся;
  - планировщик ищет следующую смену, в которую партия помещается полностью.
- Для quantity-driven операций минимальная сущность дробления:
  - `1 изд`, `1 ОС` или `1 ОК`.
- Для non-qty операций минимальная сущность дробления:
  - `1 мин`.

## Явные правила построения `plannedStartAt` и `plannedEndAt`
- Для ручных задач эти поля могут быть пустыми.
- Для auto-задач они обязательны.
- Внутри одной ячейки задачи должны выстраиваться последовательно без пересечений.
- Базовое правило:
  - первую auto-задачу в слоте начинать с `slotStartAt`;
  - каждую следующую задачу в слоте начинать с `end` предыдущей задачи в этой ячейке;
  - `plannedEndAt = plannedStartAt + plannedPartMinutes * 60_000`.
- Если задача зависит от предыдущей операции, использовать максимум из:
  - ближайшего свободного времени слота;
  - `earliestStart(nextOp)`.
- Если конец задачи выходит за пределы смены:
  - в режиме `ALLOW_SPLIT_BY_SHIFTS` разрешено создать только ту часть, которая помещается, но только если это не нарушает правило минимальной партии;
  - в режиме `SINGLE_SHIFT_ONLY` задача в этот слот не создаётся.

## Обязательное серверное API
- Добавить отдельный endpoint:
  - `POST /api/production/plan/auto`
- Запрещено перегружать существующий `POST /api/production/plan/commit` сложной логикой автопланирования.
- Новый endpoint обязан:
  - принимать все настройки автопланирования явным JSON;
  - сначала строить полный preview без сохранения;
  - валидировать маршрут, остатки, смены, участки, ёмкость, партии и задержки;
  - только после успешной валидации сохранять набор `productionShiftTasks`.
- Рекомендуемая структура действий:
  - `mode: "preview"` — только расчёт;
  - `mode: "apply"` — расчёт и сохранение.

## Обязательные клиентские сценарии
- В модалке пользователь должен иметь возможность:
  - включить автоплан;
  - поменять настройки;
  - увидеть пересчитанный preview;
  - применить план;
  - после применения сразу увидеть обновлённые ячейки сменного плана.
- Если preview показывает, что часть операций не удалось разместить, пользователь должен видеть конкретную причину:
  - нет доступных смен;
  - нет подходящего участка;
  - не помещается минимальная партия;
  - не выполнено правило задержки;
  - недостаточно передаваемого объёма от предыдущей операции.

## Обязательная диагностика
- Все новые действия должны логироваться с префиксом:
  - `[PLAN][AUTO]`
- В логах обязаны быть видны:
  - `cardId`
  - `autoPlanRunId`
  - выбранный режим
  - стартовая смена
  - горизонт
  - настройки `delayMinutes` и `transferBatchQty`
  - количество созданных задач
  - количество непокрытых операций
- Для отказов в preview или apply логировать точную причину.

## Что запрещено
- Нельзя автоматически планировать операции вне маршрутного порядка.
- Нельзя передавать на следующую операцию объём, который ещё не размещён на предыдущей.
- Нельзя игнорировать `transferBatchQty`.
- Нельзя игнорировать `delayMinutes`.
- Нельзя создавать auto-задачи без `plannedStartAt/plannedEndAt`.
- Нельзя менять формулу текущей ручной ёмкости смены.
- Нельзя молча округлять quantity-driven объём до дробной части изделия.
- Нельзя скрывать от пользователя непокрытый хвост, если он остался из-за правил партии или нехватки ёмкости.

## Критерии приёмки
- Полный автоплан по одной операции умеет разложить весь остаток по нескольким сменам.
- Полный автоплан по диапазону операций соблюдает маршрутный порядок.
- Частичный автоплан по `PARTIAL_QTY` планирует только указанное количество `изд/ОС/ОК`.
- Частичный автоплан по `PARTIAL_MINUTES` не создаёт quantity-driven задачи на дробное количество.
- Правило `transferBatchQty` реально влияет на момент старта следующей операции.
- Правило `delayMinutes` реально сдвигает старт следующей операции.
- Preview и apply дают одинаковый результат.
- Ручное планирование через старую кнопку `Сохранить` не ломается.
- Существующие `productionShiftTasks` без новых полей продолжают отображаться корректно.

Скопируй и используй этот промт для следующих задач по автопланированию, чтобы не допускать расплывчатой трактовки правил и регрессий в сменном планировщике.
