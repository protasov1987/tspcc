# Business Rules: Auth and Navigation

Этот документ фиксирует текущие рабочие правила вокруг сессии, маршрутизации,
доступов и пользовательского перехода между экранами.

Он описывает не идеальную future-модель, а текущее поведение, которое нельзя
ломать по ходу архитектурной миграции без отдельного решения.

---

## Scope

- login / logout / session restore
- protected routes
- canonical landing behavior
- browser history
- profile access
- tab-level permissions and screen visibility

---

## Actors

- Неавторизованный пользователь
- Обычный авторизованный пользователь
- Пользователь с ограниченным набором вкладок
- Администратор

---

## Current Business Invariants

- После восстановления сессии пользователь должен оказаться на маршруте,
  который указан в URL, а не на условном dashboard по умолчанию.
- Защищенный экран не должен быть показан до завершения `restoreSession()` /
  `checkAuth()`.
- Источник истины для текущего экрана в рабочем UI:
  `window.location.pathname + window.location.search`.
- Канонический домашний маршрут пользователя определяется его access level
  через `landingTab`.
- Back / Forward должны возвращать пользователя на тот экран, который
  соответствует browser history, без принудительного `F5`.
- Нельзя терять текущий рабочий маршрут только потому, что в фоне догрузились данные.
- Пользователь может открыть только свой профиль `/profile/:id`.
- Tab permissions влияют не только на меню, но и на допустимость доступа к экранам.
- Автоматический inactivity timeout является частью действующей security-модели.

---

## Critical Flows To Preserve

### Login with direct URL

- Пользователь может зайти сразу на защищенный URL.
- После логина и восстановления сессии система должна привести его именно на
  этот URL, если у него есть доступ.

### F5 on protected page

- При `F5` на защищенной странице приложение должно:
  1. восстановить сессию
  2. догрузить route-critical data
  3. открыть тот же экран

### History navigation

- При `Back` и `Forward` не должно происходить скрытого перенаправления на home.
- Исторические deep route должны открываться без ручного повторного входа.

### Permission-based landing

- Если пользователь открывает корневой или home-like маршрут, система должна
  использовать его `landingTab`, а не универсальное статичное значение.

### Profile privacy

- Попытка открыть `/profile/:id` другого пользователя не должна приводить к
  раскрытию чужого профиля.

---

## Forbidden Regressions

- Нельзя снова вводить forced redirect на dashboard при старте.
- Нельзя рендерить защищенный page content до session restore.
- Нельзя дублировать route pipeline в отдельных модулях.
- Нельзя обходить центральный router прямым переключением экранов.
- Нельзя случайно изменить список canonical routes для действующих экранов.
- Нельзя ослабить проверку profile ownership.
- Нельзя тихо отключить `landingTab`, inactivity timeout или tab access semantics.

---

## Transitional Constraints

- Текущий routing/bootstrap уже близок к target architecture и его нельзя
  деградировать ради локальной оптимизации страницы.
- Даже если внутри legacy-кода есть неудобные global functions, новый код должен
  встраиваться в существующий router/navigation flow, а не создавать второй.
- Любая миграция auth или routing должна сначала сохранять текущие URL-инварианты,
  а уже потом улучшать внутреннюю архитектуру.
