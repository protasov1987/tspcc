# Business Rules: Directories and Security

Этот документ фиксирует текущую бизнес-логику справочников, пользователей,
уровней доступа и связанных security-настроек.

---

## Scope

- подразделения / центры
- операции
- areas
- сотрудники и их привязка
- времена смен
- users
- access levels
- landing tab
- inactivity timeout
- password rules

---

## Actors

- Пользователь с правом просмотра справочников
- Пользователь с правом редактирования справочников
- Пользователь с правом управления security
- Администратор `Abyss`

---

## Current Business Invariants

- Справочники влияют на карточки, планирование и production UI, поэтому их
  редактирование не является "косметическим".
- Подразделение нельзя удалить, если к нему привязаны сотрудники.
- Подразделение нельзя удалить, если оно уже используется хотя бы в одной маршрутной карте.
- Удаление подразделения или операции не должно ломать уже созданные карточки:
  исторический текст должен оставаться читаемым.
- Тип операции нельзя менять, если это ломает уже существующее планирование.
- Операцию нельзя удалить, если она уже используется хотя бы в одной маршрутной карте.
- Времена смен являются рабочими настройками production model и не должны
  случайно сбрасываться.
- Areas участвуют в планировании и нагрузке, а значит изменение справочника
  может менять реальное производственное поведение.
- Участок нельзя удалить, если по нему есть текущее планирование или
  фактическая история выполнения в существующих production/МК данных.
- Простое упоминание участка в planning/subcontract логах без статусов
  `Годно` / `Брак` / `Задержано` удаление не блокирует.
- При удалении участка временные назначения сотрудников на него в
  `/production/schedule` должны очищаться, чтобы не возникала фантомная
  блокировка занятости.
- Пользователь должен иметь access level.
- Access level определяет:
  - видимые вкладки
  - права view/edit
  - специальные роли
  - домашний экран
  - inactivity timeout
- Edit permission по текущему смыслу всегда не уже, чем view permission.
- `Abyss` является системным администратором и должен существовать всегда.
- Пароль пользователя должен проходить серверную валидацию и uniqueness check.

---

## Critical Flows To Preserve

### Directories maintenance

- Пользователь с правами может создавать и редактировать справочники.
- Справочник не должен позволять действие, которое разрушает уже запланированную
  или связанную производственную логику.

### Employees and department assignment

- Связь сотрудника с подразделением влияет на рабочие представления и не должна
  теряться при редакторских правках.

### Access level editing

- Изменение access level должно реально менять доступные вкладки и действия.
- `landingTab` должен сохраняться и применяться как реальный home route.
- `inactivityTimeoutMinutes` должен сохраняться и применяться как часть
  действующей security-политики.

### User management

- Создание и редактирование пользователя должны оставлять пользователя
  в валидном состоянии с корректным access level.
- Смена пароля не должна обходить серверную валидацию.
- Удаление/изменение пользователя не должно разрушать системного администратора.

---

## Forbidden Regressions

- Нельзя ослабить запрет на удаление подразделения с привязанными сотрудниками.
- Нельзя разрешить удаление подразделения, которое уже используется в маршрутных картах.
- Нельзя убрать проверку на конфликт изменения типа операции с уже
  запланированными картами.
- Нельзя разрешить удаление операции, которая уже используется в маршрутных картах.
- Нельзя разрешить удаление участка, по которому уже есть история планирования
  или выполнения.
- Нельзя разорвать связь access level -> permissions -> UI/route behavior.
- Нельзя случайно сделать `Abyss` удаляемым или обычным пользователем.
- Нельзя ослабить password validation или uniqueness.
- Нельзя случайно убрать `landingTab` или inactivity timeout из рабочей модели.
- Нельзя переводить security editing в скрытые обходные UI-сценарии.

---

## Architecture Constraints

- Directories and security writes are expected to use explicit server domain
  commands, not a shared snapshot-save path.
- Directories/security persistence migration to MySQL must preserve current
  delete guards, historical text preservation, area/shift-time production
  dependencies and permission semantics.
- Security domain нельзя деградировать обратно в общий save snapshot.
- При миграции security-модели нужно сохранять current permission semantics,
  `landingTab`, `inactivityTimeoutMinutes`, password validation/uniqueness and
  `Abyss` protection, а не только поля формы.
