# Business Rules: Production and Workspace

Этот документ фиксирует текущую бизнес-логику планирования, смен,
производственного выполнения, flow-операций, очередей задержек и брака.

---

## Scope

- production schedule
- production shifts
- production plan
- workspace
- personal operations
- material issue / return
- drying
- delayed queue
- defects queue
- repair / dispose flows

---

## Actors

- Планировщик / диспетчер производства
- Руководитель с правами production edit
- Исполнитель операции
- Сотрудник с ограниченным рабочим доступом
- Сотрудник склада
- Сотрудник СКК / лаборатории
- Администратор

---

## Current Business Invariants

- Production строится вокруг неархивных `MKI`-карт с валидными операциями.
- Карта не попадает в production planning раньше, чем это разрешают ее stage и данные.
- Карта не попадает в workspace, если у нее нет реально подходящей операции на текущую смену.
- Production action не должен выполняться вне допустимого состояния flow.
- Сервер является решающим арбитром допустимости flow-действия.
- Для flow-операций действует version guard через `expectedFlowVersion`.
- Конфликт flow version должен приводить к `409`, а не к молчаливому merge.
- После конфликтного ответа рабочий контекст пользователя должен сохраниться.
- Предыдущие операции, образцы, материал, сушка, статусы ОК/ОС/изделий могут
  блокировать следующее производственное действие.
- Delayed и defect состояния являются рабочими очередями и должны сохранять
  свой бизнес-смысл при рефакторинге.
- Repair / return / dispose требуют свой набор подтверждающих действий и файлов.

---

## Visibility Rules To Preserve

- Для очереди production planning карта должна быть:
  - не архивной
  - типа `MKI`
  - с планируемыми операциями
  - в стадии `PROVIDED` или `PLANNING`
- Для очереди уже полностью запланированных карт карта должна быть `PLANNED`.
- Для workspace карта должна быть:
  - не архивной
  - типа `MKI`
  - в стадии `PLANNING` или `PLANNED`
  - с операцией, реально запланированной на текущую открытую смену
- Ограниченные роли видят только допустимую им часть операций и карточек.

---

## Critical Flows To Preserve

### Planning

- Ручное и автоматическое планирование не должны помещать карту в недопустимый
  production state.
- Частичное планирование должно сохранять семантику `PLANNING`.
- Полное планирование должно сохранять семантику `PLANNED`.

### Workspace execution

- Начать, поставить на паузу, продолжить, завершить и сбросить операцию можно
  только в допустимом состоянии.
- Нельзя завершать операцию, если сервер считает ее заблокированной.
- Personal operations должны сохранять связь с конкретной картой и конкретной операцией.

### Material issue / return / drying

- Выдача материала, возврат и сушка являются частью flow, а не отдельной
  декоративной функции UI.
- Эти действия должны по-прежнему увеличивать flow version и менять flow history.
- Нельзя завершить сушку или связанную операцию, если блокирующие условия не выполнены.

### Delayed and defect handling

- Помещение изделия или образца в `DELAYED` / `DEFECT` должно оставаться
  явным доменным действием.
- Возврат, ремонт и списание должны требовать тот контекст и те вложения,
  которые сейчас являются частью рабочего процесса.

### Repair card

- Сценарий ремонта через `МК-РЕМ` является действующей бизнес-моделью.
- Рефакторинг не должен превратить его в неявное или частично теряющее данные действие.

---

## Conflict Handling To Preserve

- `expectedFlowVersion` должен продолжать передаваться клиентом для flow writes.
- Mismatch должен продолжать возвращать `409`.
- Клиент после `409` должен:
  - остаться на текущем маршруте
  - показать понятную ошибку
  - обновить production/workspace domain slice
- Conflict нельзя "лечить" молчаливым перезаписыванием flow.

---

## Forbidden Regressions

- Нельзя открывать production action там, где сервер раньше его запрещал.
- Нельзя ослабить blocking rules по предыдущим операциям, сушке, материалу,
  образцам и quality-статусам случайным рефакторингом.
- Нельзя терять flow history.
- Нельзя превращать delayed/defect queues в простое декоративное представление
  без рабочего доменного смысла.
- Нельзя уводить пользователя с текущего workspace/workorder route только из-за conflict.
- Нельзя строить correctness production на клиентском shadow state без серверного подтверждения.

---

## Transitional Constraints

- Production сейчас является самым зрелым доменным слоем системы, и новый код
  должен ориентироваться именно на его command-style server model, а не на
  общий snapshot-save.
- Часть production UI все еще живет рядом с global arrays и legacy helper logic.
  Это можно улучшать, но нельзя при этом ломать текущие version/conflict guarantees.
- Любой перенос production logic обязан сохранять:
  - current visibility rules
  - current blocking rules
  - `expectedFlowVersion -> 409`
  - targeted refresh after conflict
