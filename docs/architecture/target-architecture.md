# Target Architecture

Этот документ описывает целевое архитектурное состояние сайта.
Это не утверждение о том, что текущая реализация уже полностью ему соответствует.

---

## Status

- Документ фиксирует target state, к которому проект должен прийти.
- Текущее состояние может частично не соответствовать этому документу.
- Любое новое изменение не должно уводить код дальше от этой модели.
- Если полное соответствие в рамках одной задачи невозможно, допустим только такой промежуточный шаг, который совместим с целевой архитектурой и не закрепляет legacy-паттерн как новую норму.

---

## Migration Rules

### MUST

- Любые изменения в `routing`, `bootstrap`, `domain writes`, `realtime`, `security`, `revision model` должны проверяться на совместимость с этой спецификацией.
- Архитектурные изменения должны вноситься маленькими шагами.
- Один шаг должен менять один логический слой или один домен.
- После каждого архитектурного шага должна быть проверка на регрессии.

### MUST NOT

- Нельзя расширять legacy-механизм, если он противоречит целевой модели, кроме минимального временного адаптера для безопасной миграции.
- Нельзя закреплять новый код на общем snapshot-save, если целевая модель требует domain writes.
- Нельзя добавлять новый bootstrap pipeline, новый независимый route pipeline или новый обходной слой управления страницами.
- Нельзя совмещать в одном изменении переход на новую write-модель с переписыванием bootstrap/router/realtime/perf-кэшей.

---

## 1. Core Model

### MUST

- Сайт должен быть SPA-приложением, где `window.location.pathname + window.location.search` является источником истины для активного экрана.
- Сервер должен быть источником истины для сохраняемых данных.
- Все критичные пользовательские изменения должны проходить через серверное подтверждение.
- Система должна быть разделена на независимые слои:
  - routing/bootstrap
  - view/UI
  - client domain actions
  - server domain API
  - persistence/revision model

### MUST NOT

- Клиент не должен считаться источником истины для бизнес-данных.
- Архитектура не должна зависеть от скрытых side-effect между bootstrap, routing, realtime и записью данных.
- Один слой не должен брать на себя ответственность другого слоя.

---

## 2. Routing

### MUST

- При прямом заходе, F5, restore session и browser history сайт должен открывать ровно тот маршрут, который указан в URL.
- `window.popstate` должен приводить к вызову `handleRoute(fullPath, { fromHistory: true, ... })`.
- Должен существовать один центральный маршрутный механизм.
- Навигационный слой должен быть идемпотентным.
- Показ и скрытие страниц должны происходить централизованно через router/page controller.

### MUST NOT

- Нельзя делать безусловный редирект на dashboard или иной домашний экран при boot.
- Нельзя навешивать параллельные независимые route-pipeline.
- Нельзя точечно управлять видимостью экранов в обход центрального роутера.
- Нельзя вызывать рендер защищенной страницы до завершения восстановления сессии.

### SHOULD

- Router should поддерживать route-local refresh без полной перестройки приложения.
- Router should логировать ключевые этапы маршрута через `[ROUTE]`.

---

## 3. Bootstrap

### MUST

- Bootstrap должен быть один и линейный.
- До завершения `restoreSession()` / `checkAuth()` разрешен только loader/overlay.
- После восстановления сессии bootstrap должен активировать маршрут из URL, а не маршрут по умолчанию.
- Boot-диагностика должна сохраняться через `[BOOT]` логи.

### MUST NOT

- Нельзя иметь второй bootstrap pipeline в другом модуле или обработчике.
- Нельзя использовать bootstrap как место для скрытых доменных переинициализаций.
- Нельзя привязывать корректность boot к realtime-каналу или background hydration.

### SHOULD

- Bootstrap should оставаться минимальным: session restore, route activation, UI shell.
- Любые ускорения should быть вторичны к стабильности boot.

---

## 4. Server Truth

### MUST

- Сервер должен подтверждать каждую критичную запись.
- Сервер должен выполнять валидацию входных данных.
- Сервер должен выполнять проверку прав доступа.
- Сервер должен выполнять проверку ревизий для критичных сущностей.
- Сервер должен возвращать точный результат доменной операции.

### MUST NOT

- Сервер не должен принимать клиентский snapshot как безусловно доверенное состояние.
- Сервер не должен silently merge конфликтующие правки.
- Сервер не должен распространять доменные эффекты вне границ конкретной операции без явной причины.

---

## 5. Revision Model

### MUST

- Каждая критично изменяемая сущность должна иметь `id` и `rev`.
- Каждая успешная запись должна увеличивать `rev` измененной сущности.
- Клиент должен передавать `expectedRev` для доменных write-операций.
- Сервер должен сравнивать `expectedRev` с фактической ревизией.
- При несовпадении сервер должен возвращать `409 Conflict`.
- Ревизия, используемая для `expectedRev`, должна принадлежать той сущности
  или доменному агрегату, который защищает write-операция.
- Для агрегированных доменных write-flow допустим coarse-grained ревизионный
  агрегат, если он инкрементится только от изменений этого домена и не зависит
  от unrelated snapshot writes.

### MUST NOT

- Нельзя заменять ревизионную модель только клиентскими guard-флагами.
- Нельзя обходить проверку ревизии в write-операции, если сущность участвует в конкурентной работе.
- Нельзя считать конфликт нормальным успехом и сохранять запись частично.
- Нельзя использовать глобальный `meta.revision` как фактическую доменную
  ревизию для конкурентных write-операций, кроме временного миграционного
  адаптера с явно запланированной заменой.

### SHOULD

- Формат ответа конфликта should включать:
  - `code`
  - `entity`
  - `id`
  - `expectedRev`
  - `actualRev`

---

## 6. Conflict Control

### MUST

- Конфликт конкурентных правок должен быть явно определенным серверным состоянием.
- После конфликта клиент должен:
  - показать понятное сообщение
  - остаться на текущем маршруте
  - обновить актуальную сущность или доменный срез
- Конфликт должен быть безопаснее, чем неявное перетирание данных.

### MUST NOT

- Нельзя молча терять чужие изменения.
- Нельзя скрывать конфликт от пользователя.
- Нельзя сбрасывать пользователя на dashboard или другой маршрут как реакцию на конфликт.
- Нельзя закрывать рабочий контекст без необходимости.

### SHOULD

- Сообщение о конфликте should быть кратким и нетехническим.
- После конфликта клиент should выполнять точечный refresh сущности, а не полную перезагрузку приложения.

---

## 7. Domain Writes

### MUST

- Все write-операции должны быть доменными и точечными.
- Каждая write-операция должна менять только нужный домен и только нужную сущность или подмножество данных.
- Для каждого домена должны существовать явные команды или endpoint'ы.
- Минимальные домены:
  - cards
  - card files
  - directories
  - users / access levels
  - production schedule
  - production shifts
  - другие критичные производственные действия

### MUST NOT

- Нельзя использовать общий snapshot-save как основную модель записи.
- Нельзя переводить сразу весь сайт на новый доменный API одним большим шагом.
- Нельзя совмещать переход на доменные API с bootstrap/perf refactor в одном изменении.

### SHOULD

- Domain API should быть command-oriented.
- Ответ доменной команды should содержать только нужный доменный результат, а не весь сайт целиком.

---

## 8. Cards Domain

### MUST

- Карточка должна быть первой классической доменной сущностью с ревизией.
- Изменение карточки должно проходить через отдельный card endpoint.
- Вложения карточки должны работать через отдельные file endpoint'ы.
- Upload/delete/resync файлов должны учитывать ревизию карточки.
- После file-операции должен возвращаться новый `cardRev`.

### MUST NOT

- Нельзя синхронизировать файлы карточки через общий snapshot сайта.
- Нельзя менять состояние карточки и файлов неатомарно без контроля расхождения.

---

## 9. Directories and Security

### MUST

- Участки, центры, времена смен, пользователи и уровни доступа должны редактироваться через отдельные доменные endpoint'ы.
- Эти сущности должны иметь ревизии, если допускают конкурентное редактирование.
- Любая запись должна включать проверку прав и ревизии.

### MUST NOT

- Нельзя отправлять весь набор справочников одним агрегированным save-запросом.
- Нельзя редактировать security-domain через обходные UI-механизмы.

---

## 10. Production Domain

### MUST

- План, расписание, смены и связанные действия должны быть отдельным доменным слоем.
- Каждое действие production должно быть явной доменной операцией.
- После production write должен обновляться только нужный production-срез.
- Production planning должен иметь собственную revision/conflict модель,
  независимую от глобального `meta.revision`.
- Для planning-side допустимы отдельные ревизии по агрегатам
  `production.schedule`, `production.plan`, `production.shifts` /
  `production.shift-close` или единая `production.planning.rev`, если она
  инкрементится только при planning mutations.
- Planning `expectedRev` должен инвалидироваться изменениями planning-domain,
  но не unrelated изменениями пользователей, сообщений, справочников или других
  snapshot-срезов.

### MUST NOT

- Нельзя строить корректность production на тяжелых локальных кэшах и pending-overlays.
- Нельзя компенсировать неудачную модель записи сложной клиентской магией.
- Нельзя переплетать production-оптимизации с bootstrap-логикой приложения.
- Нельзя считать production planning соответствующим target state, если его
  conflict control фактически построен на общем snapshot-level счетчике.

### SHOULD

- Production should сначала быть корректным и предсказуемым.
- Perf-оптимизации production should внедряться только после измерений реальной деградации.

---

## 11. Realtime

### MUST

- Realtime должен быть вспомогательным каналом уведомления и обновления.
- Система должна оставаться корректной даже при временном отсутствии realtime.
- Realtime должен работать поверх серверной модели истины, а не вместо нее.

### MUST NOT

- Нельзя делать realtime обязательным условием корректности данных.
- Нельзя заставлять bootstrap зависеть от realtime.
- Нельзя использовать realtime как замену conflict-control.

### SHOULD

- Live updates should инициировать точечный refresh сущности или домена.
- Realtime should иметь диагностические логи `[LIVE]`.
- Realtime should избегать ложного offline/noise spam.

---

## 12. Optimistic UI

### MUST

- Optimistic UI, если используется, должен быть только UX-слоем.
- Локальное optimistic состояние не должно подменять серверную истину.
- Оптимистичный шаг должен быть легко откатываемым после ответа сервера.

### MUST NOT

- Нельзя строить основную корректность сайта на сложной системе pending-state.
- Нельзя дублировать серверную модель целиком в клиентских pending-кэшах.
- Нельзя использовать optimistic слой как причину обходить server conflict response.

### SHOULD

- Допустимы короткие disabled-state, spinner, pending label.
- Optimistic UI should применяться только там, где он реально уменьшает задержку без роста архитектурного риска.

---

## 13. Performance

### MUST

- Любая оптимизация должна быть измеряемой.
- Производительность должна достигаться прежде всего:
  - точечными доменными refresh
  - ограничением избыточных рендеров
  - уменьшением объема записи
  - уменьшением объема перезапрашиваемых данных

### MUST NOT

- Нельзя переписывать bootstrap ради локальной проблемы одной страницы.
- Нельзя добавлять кэши и deferred hydration без подтвержденного bottleneck.
- Нельзя жертвовать корректностью ради perceived performance.

### SHOULD

- Perf-диагностика should быть локализованной и понятной.
- Оптимизации should вноситься после baseline measurement и перед повторным measurement.

---

## 14. Diagnostics

### MUST

- Boot, route, live и conflict-сценарии должны быть диагностируемы.
- Логи должны позволять понять:
  - где сломался boot
  - где застрял route
  - какой домен дал конфликт
  - был ли live refresh или fallback

### MUST NOT

- Диагностика не должна превращаться в постоянный шум.
- Логи не должны скрывать настоящую ошибку под множеством вторичных warning.

### SHOULD

- Использовать устойчивые префиксы:
  - `[BOOT]`
  - `[ROUTE]`
  - `[LIVE]`
  - `[DATA]`
  - `[CONFLICT]`

---

## 15. Testing

### MUST

- Критичные маршруты и бизнес-сценарии должны иметь E2E-покрытие.
- Каждый новый доменный write-механизм должен сопровождаться E2E минимум на:
  - success-path
  - conflict-path
  - route stability
  - reload/F5 stability, где применимо
- Обязательное покрытие минимум:
  - auth/bootstrap/routes
  - F5 на защищенных страницах
  - direct URL entry
  - back/forward
  - cards
  - card files
  - directories
  - production core flows
  - concurrent edit scenarios

### MUST NOT

- Нельзя вводить новый критичный write-механизм без теста конфликта.
- Нельзя менять router/bootstrap без проверки F5 и history.

---

## 16. Change Strategy

### MUST

- Архитектурные изменения должны вноситься маленькими шагами.
- Один шаг должен менять один логический слой или один домен.
- После каждого шага должна быть проверка на регрессии.

### MUST NOT

- Нельзя в одном изменении одновременно:
  - менять bootstrap
  - менять router
  - вводить новую write-модель
  - переписывать realtime
  - внедрять perf-кэши
- Нельзя переносить умные механики из нестабильной ветки целыми блоками без пересборки модели.

### SHOULD

- Порядок внедрения should быть таким:
  1. cards revisions
  2. cards conflict handling
  3. card files
  4. directories/security
  5. production
  6. optional perf improvements

---

## 17. Forbidden Final State

Сайт не должен прийти к состоянию, где одновременно или по отдельности присутствуют такие свойства:

- запись основана на большом общем snapshot
- конфликтные правки молча теряются
- bootstrap зависит от фоновой гидрации
- realtime нужен для корректности
- production требует специальных архитектурных обходов
- роутинг и рендер управляются из разных несвязанных мест
- correctness держится на костылях `pending` / local shadow state

---

## 18. Definition of Done

Можно считать, что целевая промышленная модель достигнута, если одновременно верно все ниже:

- URL полностью определяет экран.
- Bootstrap один и стабилен.
- Защищенные страницы не рендерятся до session restore.
- Write-операции доменные и точечные.
- Критичные сущности имеют `rev`.
- Конфликтные правки отрабатывают через `409`.
- Клиент корректно переживает конфликт без потери маршрута.
- Realtime не обязателен для корректности.
- Production корректен без специальных обходов ядра.
- E2E покрывают маршруты, конфликты и ключевые бизнес-сценарии.
