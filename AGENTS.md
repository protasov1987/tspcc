# AGENTS.md — Rules for Codex (tspcc-main)

Этот файл содержит обязательные архитектурные правила.
Нарушение любого пункта считается ошибкой, даже если “что-то работает”.

Актуальная архитектура проекта зафиксирована отдельно в
`docs/architecture/current-architecture.md`.
Она является обязательным контрактом для всех последующих исправлений,
доработок и рефакторингов.

`docs/architecture/migration-plan.md` является завершенной исторической записью
перехода к этой архитектуре, а не активным разрешением расширять legacy-код.
Если в коде обнаружен остаточный compatibility adapter, Codex обязан не
усиливать его и двигать изменение к removal/read-only compatibility в рамках
`docs/architecture/current-architecture.md`.

Переход persistence-слоя на MySQL 8.4 регулируется отдельно:
`docs/architecture/mysql-84-target-architecture.md` и
`docs/architecture/mysql-84-migration-plan.md`.
Любой MySQL batch обязан сохранять актуальную SPA/domain архитектуру,
business-rules и запрет на возврат critical writes к snapshot-save.

## Absolute Rules (MUST)

### A1. URL is the source of truth
- При загрузке страницы (включая F5 и прямой заход) приложение ОБЯЗАНО
  отрисовать состояние согласно:
  window.location.pathname + window.location.search.
- Запрещено показывать “dashboard по умолчанию”, если URL указывает на другую страницу.

### A2. Session-first bootstrap
- Любой роутинг, рендер страниц и инициализация UI,
  зависящие от авторизации, ДОЛЖНЫ выполняться ТОЛЬКО ПОСЛЕ:
  restoreSession() / checkAuth().
- Пока сессия не восстановлена — разрешён только loader / overlay,
  но не контент страниц.

### A3. Single bootstrap pipeline
- В приложении должен существовать ОДИН основной сценарий bootstrap
  (инициализация при загрузке страницы).
- Запрещены параллельные или дублирующие boot-процессы
  в разных файлах или событиях.

### A4. popstate is mandatory
- Обработчик window.popstate ОБЯЗАН вызывать:
  handleRoute(fullPath, { fromHistory: true, ... }).
- Back / Forward в браузере должны работать без F5 и без редиректа на dashboard.

### A5. Idempotent initialization
- initNavigation() / setupNavigation() и аналогичные setup-функции
  ОБЯЗАНЫ быть идемпотентными.
- Повторный вызов не должен:
  - вешать обработчики повторно,
  - создавать дубликаты слушателей.
- Обязателен guard-флаг вида:
  navigationSetupDone / alreadyInitialized.

Примечание:
initNavigation и setupNavigation считаются эквивалентными по смыслу;
фактическое имя функции определяется кодом файла.

### A6. No forced redirect on boot
- Запрещено делать безусловный navigate('/dashboard')
  или аналогичный редирект при старте приложения.
- Редирект допустим ТОЛЬКО если:
  - маршрут недоступен,
  - или пользователь не авторизован,
  и это решение принято внутри handleRoute.

### A7. Single page visibility control
- Показ и скрытие страниц должен происходить
  через единый управляющий механизм (роутер / showPage / render через route).
- Запрещено точечно управлять DOM (style.display и т.п.)
  в разных местах без централизованного контроля.

### A8. Boot / Route diagnostics
- Любые изменения, связанные с bootstrap или роутингом,
  ОБЯЗАНЫ сохранять или расширять существующую диагностику.
- Допустимый формат логов:
  [ROUTE] ... или [BOOT] ...
- Логи должны позволять понять:
  где именно загрузка или маршрут “завис”.

## Forbidden (MUST NOT)

F1. Нельзя менять window.location для SPA-навигации
    (кроме logout или полной перезагрузки по требованию).
F2. Нельзя вызывать render страницы ДО завершения restoreSession().
F3. Нельзя вызывать handleRoute до навешивания popstate.
F4. Нельзя добавлять новые страницы без регистрации в маршрутизаторе.
F5. Нельзя вешать обработчики меню в обход общего navigation-слоя.

## Required Pre-Commit Checklist (MUST pass)

Перед любым коммитом, затрагивающим роутинг или bootstrap:

- [ ] F5 на /dashboard работает
- [ ] F5 на /cards работает
- [ ] F5 на /cards/<id> работает
- [ ] F5 на /profile/<id> или /user/<id> работает
- [ ] Прямой заход по URL работает (без редиректа на dashboard)
- [ ] Back / Forward работает корректно
- [ ] initNavigation / setupNavigation вызывается один раз
- [ ] popstate вызывает handleRoute
- [ ] Нет скрытых редиректов при загрузке страницы

Если менялся порядок bootstrap —
ОБЯЗАТЕЛЬНО обновить docs/architecture/spa-boot.md.

## Versioning Rule (MUST)

- После каждого выполненного промта, если Codex изменил именно файлы сайта, он ОБЯЗАН увеличить версию приложения командой:
  `npm run version:bump`
- Bump версии обязателен только для изменений файлов сайта.
- К файлам сайта относятся, как минимум:
  клиентский код, серверный код, шаблоны, стили, рабочие скрипты сайта, конфигурация сайта, service worker, manifest, иконки и другие статические ресурсы сайта.
- Если изменены только документация, `.gitignore`, служебные GitHub-файлы, локальные временные файлы, IDE-артефакты или другие non-site файлы, bump версии выполнять НЕ НУЖНО.
- Файл `docs/help.html` является исключением: он считается частью сайта, а не документацией, потому что открывается из интерфейса по маршруту `/docs/help.html`.
- Перед выполнением bump версии Codex ОБЯЗАН сформулировать одно краткое и понятное описание изменений на русском языке.
- Это описание ОБЯЗАТЕЛЬНО должно передаваться в bump-команду через аргумент:
  `npm run version:bump -- --change "<краткое описание изменений>"`
- Описание изменений должно быть:
  - на русском языке,
  - кратким,
  - понятным человеку,
  - без лишних технических подробностей, если они не нужны для смысла.
- Если описание изменений не сформировано, bump версии выполнять НЕЛЬЗЯ.
- После успешного bump Codex ОБЯЗАН убедиться, что новая запись появилась в `docs/version-log.html`.
- После успешного bump Codex ОБЯЗАН убедиться, что автоматически создана локальная backup-ветка формата `версия-ДДММГГГГ-ЧЧММ`.
- Имя локальной backup-ветки ОБЯЗАТЕЛЬНО должно совпадать по версии, дате и времени с новой записью в `docs/version-log.html`.
- После успешного bump в этой локальной backup-ветке ОБЯЗАТЕЛЬНО должен автоматически создаваться локальный commit текущего состояния проекта.
- После автоматического локального backup-коммита push в GitHub выполняться НЕ ДОЛЖЕН.
- Исключение: если изменены ТОЛЬКО документационные файлы, bump версии НЕ выполняется.
- К документации относятся, как минимум:
  `docs/**`, `README.md`, `REPO_ANALYSIS.md`, `AGENTS.md`, `scripts/generate_user_manual.py` и другие файлы, которые относятся только к созданию или сопровождению документации и не влияют на работу сайта.
- Исключение из правила документации: `docs/help.html` относится к файлам сайта и ДОЛЖЕН обрабатываться как изменение сайта.
- Изменение скриптов генерации документации, скриншотов, диаграмм и сборки пользовательской инструкции НЕ считается изменением сайта и НЕ требует bump версии.
- Если вместе с документацией менялись файлы приложения, серверной логики, клиентского кода, рабочих скриптов сайта, шаблонов, стилей, конфигурации, manifest/service worker или других файлов сайта, bump выполнять ОБЯЗАТЕЛЬНО.
- Источник версии: `app-version.json`
- Строка в интерфейсе должна формироваться только из `app-version.json`, а не храниться вручную в `index.html`
- Правило инкремента:
  - `0.10.12 -> 0.10.13`
  - `0.10.99 -> 0.11.00`
  - `0.99.99 -> 1.00.00`
  - `99.99.99` при стадии `Alpha` -> `Betta 00.00.01`

## Git / VDS Publishing Rule (MUST)

- Исправлять файлы сайта напрямую на VDS НЕЛЬЗЯ.
- Любые изменения файлов сайта ДОЛЖНЫ вноситься локально, в рабочей папке проекта.
- Загружать локальные изменения сразу в `main` GitHub НЕЛЬЗЯ.
- Для каждой выгрузки на GitHub ОБЯЗАТЕЛЬНО создавать отдельную publish-ветку.
- Автоматический merge загруженной ветки в `main` РАЗРЕШЁН.
- Имя publish-ветки при загрузке на GitHub ОБЯЗАТЕЛЬНО должно иметь формат:
  `0.13.18-06042026-2159`
- Формат имени publish-ветки означает:
  `версия сайта-ДДММГГГГ-ЧЧММ`
- Версия для имени publish-ветки берётся из `app-version.json` без замены точек.
- Автоматически создаваемая после успешного `npm run version:bump` ветка формата
  `версия-ДДММГГГГ-ЧЧММ`
  считается ЛОКАЛЬНОЙ backup-веткой.
- Локальная backup-ветка создаётся только как точка быстрого локального отката.
- После автоматического создания локальной backup-ветки и локального commit
  push в GitHub выполняться НЕ ДОЛЖЕН.
- По умолчанию локальные backup-ветки пушить в GitHub НЕЛЬЗЯ.
- Если после одного или нескольких локальных backup-снимков требуется выгрузка в GitHub,
  Codex НЕ ДОЛЖЕН пушить все локальные backup-ветки подряд.
- Codex ОБЯЗАН выбрать одно итоговое состояние сайта,
  которое действительно должно попасть в GitHub и затем на VDS.
- Publish-ветка создаётся осознанно перед выгрузкой:
  - либо от текущего рабочего состояния,
  - либо от выбранной локальной backup-ветки.
- Publish-ветка и локальная backup-ветка считаются РАЗНЫМИ ветками по назначению,
  даже если их имена имеют одинаковый формат.
- Имя publish-ветки МОЖЕТ иметь тот же формат, что и backup-ветка:
  `версия-ДДММГГГГ-ЧЧММ`
- Одинаковый формат имени НЕ означает,
  что локальная backup-ветка автоматически становится publish-веткой.
- Статус ветки определяется не шаблоном имени, а её назначением:
  - backup-ветка: для локального отката;
  - publish-ветка: для выгрузки в GitHub и последующего deploy на VDS.
- В GitHub должна выгружаться только publish-ветка,
  а не весь набор локальных backup-веток.
- Если текущее состояние сайта уже является нужным итоговым состоянием,
  Codex ДОЛЖЕН создать одну publish-ветку и пушить только её.
- Если существует несколько локальных backup-веток,
  Codex НЕ ДОЛЖЕН пушить их все без явной необходимости.
- Перед push в GitHub Codex ОБЯЗАН явно понимать и указывать,
  какая именно ветка является publish-веткой,
  и что именно она является целевым состоянием для публикации.
- Если пользователь явно не требует иного,
  локальные backup-ветки должны оставаться только локальными.
- Файлы сайта на VDS ОБЯЗАТЕЛЬНО должны попадать только из GitHub через deploy.
- Ручная замена, копирование или правка файлов сайта на VDS вне deploy запрещены.

## MCP Usage Rules (MUST)

MCP использовать только при точной необходимости.

Если задачу можно корректно выполнить по локальным файлам проекта,
уже доступному контексту или обычным shell-командам, MCP использовать
ЗАПРЕЩЕНО.

Перед каждым вызовом MCP Codex ОБЯЗАН понимать:
- какую конкретную информацию он хочет получить;
- почему без MCP эта информация недоступна или ненадежна;
- что вызов MCP не создаст лишних изменений или побочных эффектов.

### MCP: Forbidden

- Нельзя вызывать MCP "на всякий случай".
- Нельзя использовать MCP для проверки очевидных вещей, которые можно узнать
  из локального репозитория.
- Нельзя дублировать один и тот же MCP-запрос без новой причины.
- Нельзя использовать write-capable MCP-действия без прямого запроса пользователя.
- Нельзя отправлять в MCP секреты, токены, пароли, приватные ключи или лишние
  фрагменты кода.
- Нельзя использовать GitHub MCP для изменения удаленного репозитория, если
  пользователь явно не попросил это сделать.
- Нельзя использовать browser MCP вместо локальных тестов, если задача не
  требует проверки UI в браузере.

### MCP: `mysql-readonly`

Использовать только для read-only проверки MySQL:
- посмотреть список баз, таблиц, схем;
- выполнить `SELECT`, `SHOW`, `DESCRIBE`, `EXPLAIN`;
- сверить фактическую структуру MySQL 8.4 с миграциями, документацией или кодом.

Запрещено:
- использовать для любых записей;
- использовать вместо анализа миграций, если достаточно локальных файлов;
- делать тяжелые запросы без `LIMIT` или без понятной причины.

### MCP: `github`

Использовать только когда нужна информация из GitHub, которой нет локально:
- проверить PR, issue, release, commit, remote file;
- получить данные авторизации GitHub;
- выполнить точечный secret scan перед публикацией, если есть конкретный
  diff/content.

Запрещено:
- создавать или обновлять файлы в GitHub без прямого указания пользователя;
- комментировать issue/PR без прямого указания пользователя;
- создавать PR или назначать Copilot без прямого указания пользователя;
- использовать GitHub MCP вместо локального `git`, если локальный репозиторий
  содержит нужные данные.

### MCP: `context7`

Использовать только для актуальной документации библиотек и фреймворков:
- когда API мог измениться;
- когда нужна точная современная сигнатура, пример или best practice;
- когда локальный код не отвечает на вопрос.

Запрещено:
- использовать для общих знаний;
- использовать без предварительного выбора точной библиотеки;
- полагаться на Context7 вместо официальной документации, если вопрос критичный
  и есть официальный источник.

### MCP: `openaiDeveloperDocs`

Использовать для OpenAI API, SDK, Codex, ChatGPT Apps и связанных продуктов:
- при вопросах о моделях, API, параметрах, endpoints;
- при необходимости актуальной официальной документации;
- при миграциях между OpenAI API.

Запрещено:
- использовать сторонние источники вместо этого MCP для OpenAI-вопросов, если
  официальная документация достаточна;
- отвечать по памяти на нестабильные вопросы об OpenAI API без проверки.

### MCP: `playwright`

Использовать только для проверки поведения сайта в браузере:
- роутинг, SPA-навигация, Back/Forward, F5;
- визуальная проверка UI;
- проверка кликов, форм, accessibility snapshot;
- проверка сетевых запросов во время пользовательского сценария.

Запрещено:
- использовать вместо unit/integration тестов, если браузерная проверка не нужна;
- делать screenshots без необходимости;
- менять данные через UI без понимания последствий.

### MCP: `chrome-devtools`

Использовать только для диагностики открытой браузерной страницы:
- console errors;
- network requests;
- snapshots;
- Lighthouse;
- точечная проверка DOM/UI.

Запрещено:
- использовать параллельно с Playwright для одной и той же простой проверки без
  причины;
- запускать Lighthouse без необходимости;
- использовать для действий, которые проще и надежнее выполнить через Playwright.

### MCP: Priority

Порядок выбора источников:
1. Локальные файлы и `rg`.
2. Локальные shell-команды.
3. Локальные тесты проекта.
4. MCP только если предыдущих способов недостаточно.
5. Web search только если MCP/локальные источники не подходят или нужна внешняя
   актуальная информация.

Перед использованием MCP Codex ОБЯЗАН кратко сообщить, зачем он его вызывает.
После использования MCP Codex ОБЯЗАН учитывать результат как внешний источник
данных и не делать лишних дополнительных вызовов без новой причины.
